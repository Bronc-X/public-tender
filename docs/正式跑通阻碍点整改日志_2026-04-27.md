# 正式跑通阻碍点整改日志

记录时间：2026-04-27  
项目：智能招标智能体 / bid_data_management  
范围：只记录诊断结论和修改建议，不包含已实施代码改动。  
目标：把“离正式跑通还差什么”拆成可执行、可验证的整改清单。

## 总体判断

当前问题不只是“最后无法生成文件”。最后 Word 导出失败是末端症状，前面还有服务启动、前端构建、数据库路径、文件路径、Step6 状态一致性等阻碍。

正式跑通的最短路径是：

1. 先让当前机器能启动当前源码。
2. 再让前端生产构建通过。
3. 再清理数据库里的旧机器文件路径。
4. 再修 Step6 生成、审核、导出、下载闭环。
5. 最后用浏览器端到端跑一次商务标流程。

## P0 阻碍：服务无法稳定从当前源码启动

### P0-1 后端数据库路径写死到旧机器

症状：

后端入口硬编码数据库路径：

```go
dbPath := "/Users/raoyi/.openclaw/workspace/hudi/bid_data_management/backend_go/data/app.db"
```

位置：

`backend_go/cmd/server/main.go:18`

证据：

当前机器实际项目路径是：

```text
/Users/broncin/Desktop/project/智能招标智能体/bid_data_management
```

旧路径不存在：

```text
/Users/raoyi/.openclaw/workspace/hudi/bid_data_management/backend_go/data/app.db
```

影响：

即使重新编译成功，后端也会尝试连接旧机器数据库，服务启动失败或读错环境数据。用户看到的是页面接口全挂，或者数据和当前项目不一致。

建议修改：

把数据库路径改成环境变量优先，默认使用项目内相对路径：

```go
dbPath := os.Getenv("APP_DB_PATH")
if dbPath == "" {

}
```

```bash

```

通过标准：

```json

```

### P1-2 Step6 approve 接口前端调用了，但后端没有路由

症状：

前端调用：

```text
POST /api/bid-projects/:id/step6/slots/:slotId/approve
```

位置：

`frontend/src/components/CommerceChapterGenerationPanel.tsx:278`

但后端只注册了：

```text

```

影响：

用户在前端点击“通过”时，后端不会持久化审核状态。前端 `.catch(() => {})` 又把错误吞掉，用户以为通过了，其实状态没有保存。

建议修改：

二选一：

1. 后端补 `approve` 路由，更新 `step6_payload_json` 中对应 slot。
2. 前端去掉伪 approve 接口调用，只保留本地编辑，并在导出前明确提示“未持久化”。

推荐：

补后端 `approve` 路由。因为正式导出应该由后端状态说了算。

验证命令：

```bash
curl -X POST http://localhost:8081/api/bid-projects/<project_id>/step6/slots/<slot_id>/approve \
  -H "Content-Type: application/json" \
  -d '{"ai_suggested_value":"测试值"}'
```

通过标准：

接口返回 200，刷新 Step6 payload 后对应 slot 状态仍是 `approved`。

## P1 阻碍：Word 替换不可靠

### P1-6 CleanDocxTags 还是空实现

症状：

`CleanDocxTags` 当前直接返回原 XML：

```go
return xmlContent
```

位置：

`backend_go/internal/service/docx_template.go:57`

影响：

Word 经常把 `{{字段}}` 拆成多个 `<w:t>` 标签。当前实现无法处理这种情况。用户会看到“文件生成了，但是占位符没被替换”。

建议修改：

实现真正的占位符归并逻辑，至少支持：

```xml
<w:t>{</w:t><w:t>{</w:t><w:t>项目经理</w:t><w:t>}</w:t><w:t>}</w:t>
```

归并成：

```text
{{项目经理}}
```

验证标准：

新增一个包含拆分占位符的 `.docx` fixture，导出后占位符能被替换。

### P1-7 替换值没有 XML escape

症状：

导出时直接：

```go
strings.ReplaceAll(strContent, k, v)
```

影响：

如果填充值包含 `&`、`<`、`>`，Word XML 会损坏。用户下载后可能打不开文档。

建议修改：

写入 Word XML 前对文本做 XML escape。

验证标准：

用包含 `A&B <测试>` 的字段导出，Word 能正常打开，内容正确显示。
