# 目录生成提速方案（程序员执行清单版）

> 目标：目录生成速度从 10+ 分钟压缩到 2~3 分钟

---

## 一、核心思路

把目录生成拆成两段：
- **第一段：快速出首版目录（2 分钟内）**
- **第二段：后置精修校验（用户确认后再跑）**

---

## 二、第一段：快速首版生成

### 只保留 3 步

#### 步骤 1：轻量 facts 提取

##### 新增函数
`ExtractOutlineFactsFast`

##### 位置
`backend_go/internal/service/tender_digitization.go`

##### 和现有 `ExtractOutlineFacts` 的区别
- 不分章节
- 不分视角（不跑 audit_view / scoring_view / technical_view / cost_view 四个视角）
- 只做一次整体提取
- 只抽：评分项、强制项、特殊技术点
- 超时设为 60 秒
- 不构建 requirement register
- 不做缺项回扫补抽

##### 输入
招标文件全文（截断到 50000 字符）

##### 输出
`*FactExtractResult`（和现有结构一样）

---

#### 步骤 2：目录直生

##### 函数
现有的 `GenerateOutlineDirectly`，不改

##### 输入
- 招标文件原文
- 完整提示词
- 路线规划
- 少量高权重 facts（来自步骤 1）

---

#### 步骤 3：必出项硬校验 + 补章

##### 函数
现有的 `PatchMissingMandatoryChaptersWithLoop`，不改

##### 逻辑
纯本地代码，不调模型，耗时约 1 秒

---

### 新增快速模式入口

##### 新增函数
`RunOutlinePipelineFast`

##### 位置
`backend_go/internal/service/step4_coordinator.go`

##### 伪代码
```go
func RunOutlinePipelineFast(p *OutlinePipelineInput) {
    // 1. 轻量 facts 提取（一次调用，约 30~60 秒）
    facts, err := d.ExtractOutlineFactsFast(ctx, id, tenderContent)
    
    // 2. 目录直生（一次调用，约 60~90 秒）
    outline, err := d.GenerateOutlineDirectly(ctx, id, tenderContent, facts, nil, "", routeName, projectType, profession)
    
    // 3. 必出项硬校验 + 补章（纯本地，约 1 秒）
    mandatoryChapters := ExtractMandatoryChapters(facts)
    outline, _ = PatchMissingMandatoryChaptersWithLoop(outline, mandatoryChapters, 3)
    
    // 4. 落库
    // 保存到 tech_bid_chapter_plans
    // 更新 project 状态为 outline_ready
}
```

---

### 快速模式下跳过的步骤

| 跳过的步骤 | 对应函数 | 理由 |
|-----------|---------|------|
| requirement register 构建 | `BuildRequirementRegister` | 重型，后置 |
| 缺项回扫补抽 | `RescanMissingRequirements` | 重型，后置 |
| 冲突审计 | `DetectTenderConflicts` | 重型，后置 |
| fact → outline 映射 | `BuildFactToOutlineMappings` | 重型，后置 |
| 覆盖率审计 | `AuditOutlineCoverage` | 重型，后置 |
| 优化补全 | `OptimizeOutlineByCoverage` | 重型，后置 |

---

## 三、第二段：后置精修校验

### 触发时机
用户看完首版目录，点击"精修校验"按钮后触发

### 执行内容
就是现在 `RunOutlinePipeline` 里跳过的那些步骤：
1. requirement register 构建
2. 冲突审计
3. 覆盖率审计
4. 如果需要，优化补全

### 执行方式
后台异步执行，不阻塞用户

---

## 四、前端改动

### 需要加的
1. 默认走快速模式生成目录
2. 目录生成后直接展示给用户
3. 加一个"精修校验"按钮
4. 点击后触发后置精修
5. 精修完成后更新目录状态

### 不需要改的
- 目录展示组件
- 目录编辑交互
- 其他步骤的前端

---

## 五、后端 API 改动

### 方案 A：新增快速生成接口
新增一个 API：
```
POST /api/tech-bid/projects/:id/outline/generate-fast
```
走 `RunOutlinePipelineFast`

现有接口保持不变，走完整流程

### 方案 B：现有接口加参数
在现有生成接口上加一个参数：
```json
{ "mode": "fast" }
```
- `fast`：走快速模式
- `full`（默认）：走完整流程

**建议用方案 B，改动更小。**

## 六、实施顺序

### 第一步
新增 `ExtractOutlineFactsFast` 函数

### 第二步
新增 `RunOutlinePipelineFast` 函数

### 第三步
后端 API 支持 `mode=fast` 参数

### 第四步
前端默认走快速模式

### 第五步
前端加"精修校验"按钮

### 第六步
后置精修异步执行

---

## 七、验收标准

1. 快速模式下，目录生成 **2~3 分钟内完成**
2. 首版目录质量不明显低于当前版本
3. 必出项硬校验仍然生效
4. 用户可以选择"精修校验"来补全质量
5. 精修校验不阻塞用户操作
6. 现有的完整模式仍然可用，不受影响

---

## 八、不改的部分

| 不改 | 理由 |
|------|------|
| `GenerateOutlineDirectly` | 已优化，保持不变 |
| 必出项硬校验 | 纯本地，不影响速度 |
| 骨架冻结 / drift 校验 | 纯本地，不影响速度 |
| 提示词 | 已优化，保持不变 |
| 完整模式 `RunOutlinePipeline` | 保留作为精修路径 |

---

## 九、一句话总结

**新增一个"快速模式"：只做轻量 facts 提取 + 目录直生 + 必出项校验，2 分钟出首版；重型校验全部后置，用户想精修时再跑。**
