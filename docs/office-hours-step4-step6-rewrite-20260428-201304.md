## Problem Statement

用户要的不是“Step4 分数看起来更高”，而是技术标从招标文件进入目录生成，再进入最终文档输出的全流程真的通路。

当前真实项目 `24f46df0-1dd1-4b4f-90de-24869cea62da` 没有通：

- `current_step = outline_generation`
- `current_step_status = success`
- `step4_status = audit_ready`
- `final_decision = BLOCK`
- `can_enter_content_generation = 0`
- `coverage_score = 45.0`
- `full_response_rate = 87.5`
- `step6_status = idle`
- `tech_bid_outputs` 没有该项目输出记录

这说明 Step6 返回最终文档这一步没有正常进入。之前商务标项目 `1295a2e4-89a0-458c-84ac-3d02433945a4` 的 Step6 成功，不能证明技术标项目成功。它走的是 `bid_projects` 与 `/api/bid-projects/:id/step6/...`，不是当前技术标项目的 `tech_bid_projects` 链路。

## Demand Evidence

这不是假设问题，已经有运行证据：

- Run 37：走到审计，但结果是 `REVISE`。出现重复“第一章”，要求 ID 与真实业务要求不稳定。
- Run 38：结果 `BLOCK`。优化后 ID 丢失，覆盖率坍塌。
- Run 39：结果 `BLOCK`。结构覆盖看似可到 100，但持久化后 `requirement_ids_json` 变成 null。
- Run 40：修复持久化后仍 `BLOCK`。`coverage_score = 45.0`，`full_response_rate = 87.5`，高优先级 `ms_002` 缺失。大量技术节点挂的是 `ms_01` 这类泛化元规则，不是真正的业务响应。

最要命的是：现在系统把“有 ID”误当成“已响应”。这会让测试结果看起来在前进，但用户拿不到可信的最终技术标文档。

## Status Quo

当前流程大致是：

1. `TechBidProjectHandler.SelectRoute` 启动 `RunOutlinePipeline`。
2. `BuildRequirementRegister` 依赖分章节 LLM 扫描生成要求登记表。
3. `ValidateFullRequirementResponse` 检查目录是否覆盖登记表。
4. `RunOutlinePipeline` 把目录写入 `tech_bid_chapter_plans`。
5. `ConfirmStep` 只有在 Step4 PASS 或放行时允许进入 `content_generation`。
6. Step6 成熟实现主要在 `BidProjectHandler` 与 `Step6ExporterService`，面向旧商务标 `bid_projects`。

现状成本：

- 一次完整 Step4 真实运行约 11 到 14 分钟。
- 慢点集中在真实 LLM 事实提取和要求扫描，DashScope 多次出现 timeout 或 EOF 重试。
- 多次重复跑会消耗大量时间，却不能改变根因。

## Premises

1. Step4 的真相层必须从“ID 是否存在”改成“业务要求是否被技术目录明确响应”。
2. 元规则、项目概况、评分条款、废标条款、施工技术要求不能混在同一个无类型 ID 池里。
3. Step6 必须有技术标自己的入口、状态、输出表和下载闭环，不能用商务标成功结果冒充。
4. 在设计通过前，不继续做小补丁。否则会继续出现“局部测试过了，全流程还是不通”。
5. API key 只能作为运行时配置，不能写入代码、测试 fixture 或文档明文。

## Approaches Considered

### Approach A: 最小修补版

Summary: 保留现有 Step4 主流程，只加强 `BuildRequirementRegister`、`ValidateFullRequirementResponse` 和 `ConfirmStep` 的若干判断，再补一个技术标 Step6 下载入口。

Effort: M

Risk: Medium

Pros:

- 改动文件少，能最快看到 Step4 从 BLOCK 变 PASS 的机会。
- 复用已有 `RunOutlinePipeline`、`tech_bid_chapter_plans`、审计表和前端状态。
- 对当前 UI 影响小，适合短期止血。

Cons:

- 仍然把关键真相交给 LLM 分章节扫描，慢且不稳定。
- 容易继续把 `ms_01` 这类元规则当成技术响应。
- Step6 能导出不代表内容可信，可能只是把不可信的目录装进文档。

Reuses:

- `requirement_full_response.go`
- `outline_strong_mapping.go`
- `step4_coordinator.go`
- `step6_exporter_service.go`

### Approach B: 推荐方案，要求优先的有界重写

Summary: 重写 Step4 的真相层和关口判断，但不重写整个系统。先建立稳定的要求登记表，再让目录生成和 Step6 都围绕这张表工作。

Effort: L

Risk: Medium

Pros:

- 正面解决根因：业务要求、元规则、项目概况分层，不再靠“有 ID”假装通过。
- 可以把 LLM 从“唯一真相来源”降级为“补充抽取和润色”，减少 14 分钟反复跑的浪费。
- Step6 会基于通过审计的 `tech_bid_chapter_plans` 和章节内容生成技术标输出，链路更清楚。

Cons:

- 需要改动 Step4 核心服务，测试必须补齐，否则风险比小修高。
- 会触碰既有数据模型的边界，需要兼容旧表字段，避免一次性迁移过大。
- 第一版可能仍依赖一个真实 LLM 冒烟用例，运行时间不会立刻变成秒级。

Reuses:

- `TenderDigitizationService.SplitDocumentToChapters`
- `tech_bid_requirement_register`
- `tech_bid_outline_facts`
- `tech_bid_fact_mappings`
- `tech_bid_requirement_response_checks`
- `tech_bid_chapter_plans`
- `tech_bid_outputs`
- `Step6ExporterService` 的 DOCX 安全替换能力

### Approach C: 旁路直出版

Summary: 暂时绕开 Step4 PASS，用招标文件、项目画像、目录草案直接生成技术标 Word，再用事后审计标红风险。

Effort: M

Risk: High

Pros:

- 最快能让用户下载到一个 Word 文件。
- 可以用于演示“最终导出和下载”这条 UI 链路。
- 对 Step4 的复杂审计依赖最低。

Cons:

- 会牺牲可信度。用户要的是可投标文档，不只是一个可下载文件。
- 容易绕过废标项、评分项、高优先级响应要求。
- 一旦用户误以为“能下载等于可用”，后果比现在 BLOCK 更糟。

Reuses:

- `GenerateContentForChapter`
- `Step6ExporterService.ExecuteSafeWordExport`
- `tech_bid_outputs`

## Recommended Approach

选择 Approach B。

原因很简单：现在的问题不是最后一个按钮坏了，是前面的“可进入最后一步”的判定没有可信真相层。先把真相层写对，再接 Step6。否则只是更快地产出不可信文档。

这不是重写整个应用。边界必须收住：

- 不重写前端工作台。
- 不重写商务标 `bid_projects` Step6。
- 不重写文件上传、OCR、DocMind 解析。
- 不做新的大模型平台抽象。
- 不把用户给的 API key 写进仓库。

## Proposed Architecture

### 1. Requirement Registry v2

新增或扩展要求登记表构建逻辑，核心字段必须能表达：

- `requirement_id`
- `requirement_kind`: `business_requirement`、`scoring_item`、`mandatory_clause`、`project_metadata`、`meta_rule`
- `source_section`
- `source_text`
- `priority`
- `must_be_explicit`
- `expected_response_level`
- `response_tier`
- `summary`

关键规则：

- 技术章节只能用 `business_requirement`、`scoring_item`、`mandatory_clause` 作为通过依据。
- `project_metadata` 只允许落在工程概况、编制依据、项目总述等概况章节。
- `meta_rule` 可以作为背景约束，但不能计入 Step4 PASS 的业务覆盖。

### 2. Deterministic Prepass

在 LLM 扫描前先做确定性预扫描：

- 从“评标办法”“技术标准和要求”“废标条款”“施工组织设计评分标准”等章节抽取候选要求。
- 用章节标题、条款编号、关键词、冒号/分号结构建立稳定 ID。
- 对明显的工期、质量、安全、文明施工、环保、进度、资源配置、重难点、专项方案条款生成业务要求。

LLM 只负责补漏和摘要，不再负责决定整个登记表是否存在。

### 3. Coverage Gate v2

替换当前“挂 ID 即响应”的弱判断：

- 高优先级业务要求必须 100% 覆盖。
- `mandatory_clause` 缺失直接 BLOCK。
- 技术章节中如果主要挂 `meta_rule`，不算通过。
- 同一个泛化 ID 覆盖过多章节时降权或报警。
- 章节标题必须与要求语义有可解释关系，不能只靠数组里有 ID。

输出必须写入 `tech_bid_requirement_response_checks`，让前端和日志能解释为什么 PASS、REVISE 或 BLOCK。

### 4. Outline Persistence Contract

`tech_bid_chapter_plans.requirement_ids_json` 只保存有效业务要求 ID。

持久化前做三件事：

- 去掉无效 ID。
- 去掉不允许进入技术章节的 `meta_rule`。
- 如果节点没有业务要求，明确保存空数组，不保存 null。

这能避免后续 Step6 和前端状态被 null、字符串数组、对象数组混用拖垮。

### 5. Tech Step6 Adapter

给技术标建立自己的 Step6 入口，不能复用旧商务标路由语义：

- `GET /api/tech-bid/projects/:id/step6/payload`
- `POST /api/tech-bid/projects/:id/step6/generate`
- `POST /api/tech-bid/projects/:id/step6/export`
- `GET /api/tech-bid/projects/:id/step6/download`

Step6 入口规则：

- 如果 `final_decision != PASS` 且未显式放行，返回 `STEP4_NOT_PASSED`，不生成文档。
- 如果 Step4 PASS，则从 `tech_bid_chapter_plans` 和 `tech_bid_chapter_contents` 组装 payload。
- 导出成功后写入 `tech_bid_outputs`，`output_type = technical_word`。
- 下载接口必须返回 DOCX MIME 和非零文件大小。

### 6. Runtime Policy

完整真实 LLM 跑一次即可，不再无目的重复跑。

顺序：

1. 先跑确定性单元测试。
2. 再跑后端全量测试。
3. 再跑前端构建。
4. 再启动本地服务做 API 冒烟。
5. 最后用当前真实项目跑一次 Step4 到 Step6。

## Success Criteria

### Unit level

- `go test ./...` 通过。
- 新增测试覆盖：
  - 元规则不能让技术章节通过。
  - 高优先级业务要求缺失时 BLOCK。
  - `requirement_ids_json` 不再写 null。
  - 技术 Step6 在 Step4 未 PASS 时拒绝生成。
  - 技术 Step6 在 Step4 PASS 且有章节内容时写入 `tech_bid_outputs`。

### Build level

- 后端可启动：
  - `APP_DB_PATH=data/app.db PORT=8081 go run ./cmd/server/main.go`
  - `/health` 返回 `{"status":"up"}`
- 前端生产构建通过：
  - `npm run build`

### API smoke level

对真实项目 `24f46df0-1dd1-4b4f-90de-24869cea62da`：

- Step4 运行结束后必须满足：
  - `final_decision = PASS`
  - `can_enter_content_generation = 1`
  - `step4_gate_result = PASS`
  - 高优先级业务要求缺失数为 0
- Step6 运行结束后必须满足：
  - `step6_status = success`
  - `step6_payload_json` 非空
  - `tech_bid_outputs` 有 `technical_word` 输出
  - 下载接口 HTTP 200
  - DOCX 文件大小非零

### Browser smoke level

用浏览器验证并截图：

- 技术标项目页面能进入最终输出态。
- 页面展示下载入口。
- 点击下载不报错。
- 刷新后仍保持成功态。
- 控制台没有新增 error。

### Document quality level

导出的 DOCX 至少可检索到：

- 项目名称或招标编号。
- 施工组织设计相关章节。
- 与评分或强制条款对应的内容片段。
- 不出现纯占位符文案。

## What I Noticed

- 你关心的是“全流程是否通路，是否冒烟，输出结果是否 OK”，这是正确的交付标准。
- 你对“跑了一个多小时到底结果是什么”的追问非常关键。这个系统的问题正是测试成本高、反馈不清、状态没有对齐。
