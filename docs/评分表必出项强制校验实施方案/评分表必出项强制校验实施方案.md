# 评分表必出项强制校验 — 技术实施方案

> CTO 出品 | 2026-04-10

---

## 一、问题

评分表中明确独立列出的得分项（如"环境保护管理体系和措施"），被模型合并到其他章节里，导致评审时判定为"缺漏项"，直接扣 6 分。

根因：目前的目录生成流程，**没有把评分表里的独立项当成硬约束**，模型可以自由决定合并或拆分。

---

## 二、解决思路

### 整体逻辑

```
Facts 提取阶段
    ↓
从 facts 中识别出"评分表必出项"（A 类）
    ↓
目录生成时，把必出项作为硬约束单独喂给模型
    ↓
生成后做硬校验：必出项是否全部独立成章
    ↓
缺项 → 自动补章
```

---

## 三、代码修改

### 改动 1：新增"必出项提取"函数

**文件**：`backend_go/internal/service/tender_digitization.go`

**新增函数**：`ExtractMandatoryChapters(facts *FactExtractResult) []string`

**功能**：从 facts 中筛选出必须独立成章的条目

**筛选规则**：
- facts.ScoreItems 中，名称包含以下关键词的，必须独立成章：
  - "施工方案"或"技术措施"
  - "质量管理"
  - "安全文明"或"安全管理"
  - "环境保护"
  - "工期保证"或"工期"
  - "资源配备"或"资源"
  - "施工总平面"或"平面布置"
  - "项目管理机构"
- facts.MandatorySpecs 中，priority 为 high 且名称明确指向独立章节的

**返回**：必出项名称列表，如 `["施工方案和技术措施", "质量管理体系与措施", "安全文明施工管理体系与措施", "环境保护管理体系与措施", ...]`

---

### 改动 2：目录生成时，把必出项单独喂给模型

**文件**：`backend_go/internal/service/tender_digitization.go` — `GenerateOutlineDirectly` 函数

**改动位置**：在构建 `dynamicInstruction` 时

**改动内容**：

在现有的 `keyFactsSummary` 之外，再加一段"必出项清单"：

```go
mandatoryChapters := ExtractMandatoryChapters(facts)
mandatoryText := ""
if len(mandatoryChapters) > 0 {
    mandatoryText = "【强制独立成章项（不得合并到其他章节）】\n"
    for i, ch := range mandatoryChapters {
        mandatoryText += fmt.Sprintf("%d. %s\n", i+1, ch)
    }
}
```

然后把 `mandatoryText` 插入 `dynamicInstruction` 中，放在 `keyFactsSummary` 前面：

```go
dynamicInstruction := fmt.Sprintf(`### 以下为评分表中必须独立成章的条目（硬性约束，不得合并）
%s

### 以下为从招标文件中预抽取的评分项和强制项摘要（仅供辅助参考）
%s

请严格按上述提示词要求和招标文件内容生成三级目录。只返回 JSON 数组。`, mandatoryText, keyFactsSummary)
```

### 改动 3：生成后做"必出项硬校验"

**文件**：`backend_go/internal/service/tender_digitization.go`

**新增函数**：`ValidateMandatoryChapters(outline []map[string]interface{}, mandatoryChapters []string) (missing []string, ok bool)`

**功能**：
- 遍历 outline 的一级章名称
- 逐一检查 mandatoryChapters 中的每一项，是否在一级章名称中有语义匹配
- 匹配规则：一级章名称包含必出项的核心关键词即算命中
- 返回缺失项列表

**匹配示例**：
- 必出项"环境保护管理体系与措施" → 一级章名称包含"环境保护" → 命中
- 必出项"安全文明施工管理体系与措施" → 一级章名称包含"安全文明" → 命中

---

### 改动 4：缺项自动补章

**文件**：`backend_go/internal/service/tender_digitization.go`

**新增函数**：`PatchMissingMandatoryChapters(outline []map[string]interface{}, missing []string) []map[string]interface{}`

**功能**：
- 对每个缺失的必出项，在 outline 末尾追加一个一级章
- 章名使用必出项名称
- 下面挂一个占位二级节，如"第一节 总体措施"
- 下面挂一个占位三级小节，如"一、待补充"

**目的**：
- 确保骨架完整
- 后续正文撰写阶段再填充内容

---

### 改动 5：在主链路中串联

**文件**：`backend_go/internal/service/step4_coordinator.go` 和 `backend_go/internal/handler/tech_bid_project.go`

**改动位置**：目录生成完成后、进入审计之前

**改动逻辑**：

```go
// === 必出项硬校验 ===
mandatoryChapters := ExtractMandatoryChapters(facts)
missing, ok := ValidateMandatoryChapters(draftOutline, mandatoryChapters)
if !ok {
    log.Printf("[Step4] 必出项缺失: %v，自动补章", missing)
    draftOutline = PatchMissingMandatoryChapters(draftOutline, missing)
}
```

## 四、提示词修改

### 修改位置
后台提示词 `tech_bid_outline_direct_generation` 的 **指令约束（System Role）**

### 在现有内容末尾追加

```
七、评分表独立项必须独立成章
如果评分标准中明确列出了以下独立评分维度：
- 施工方案和技术措施
- 质量管理体系与措施
- 安全文明施工管理体系与措施
- 环境保护管理体系与措施
- 工期保证措施
- 资源配备计划
- 施工总平面布置
- 项目管理机构配置

则每一项必须在目录中独立成章（一级章），不得合并到其他章节中。
评委在一级目录中找不到对应章名，即判定为"缺漏项"，直接扣分。
因此：宁可多一章，不可少一章。
```

## 六、验收标准

1. 评分表中的独立项，全部出现在一级章中
2. 如果模型漏掉了某个必出项，系统自动补章
3. "环境保护"不再被合并到"安全文明"里
4. 提示词明确告诉模型"宁可多一章，不可少一章"

---

## 七、风险评估

| 风险 | 等级 | 应对 |
|------|------|------|
| 必出项关键词匹配不准 | 低 | 用宽松匹配（包含核心关键词即命中） |
| 自动补章的占位内容太空 | 低 | 后续正文撰写阶段填充 |
| 不同招标文件评分项名称差异大 | 中 | 必出项清单从 facts 动态提取，不写死 |

---

## 八、一句话总结

**把评分表里的独立项，从"建议"升级成"硬约束"：提示词里写死、代码里校验、缺了自动补。**
