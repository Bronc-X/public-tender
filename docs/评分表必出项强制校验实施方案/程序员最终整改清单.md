# 评分表必出项强制校验 — 程序员最终整改清单

> 目标：彻底避免“环境保护”等评分表独立项被合并、漏掉

---

## 一、总原则

1. **评分表里的独立项，必须独立成章**
2. **宁可多一章，不可少一章**
3. **不要只靠提示词，必须有代码硬校验**
4. **必出项来源要尽量前置，不能只靠 facts 反推**
5. **补章后要再次校验，不能补完就放行**

---

## 二、要做的事情

### 任务 1：新增“评分表独立项”抽取逻辑

#### 目的
从评分表原文中，提前识别出必须独立成章的条目，形成 `mandatory_chapters`。

#### 建议位置
`backend_go/internal/service/tender_digitization.go`

#### 要求
- 不是只从 `facts` 里猜
- 尽量从评分标准 / 综合评分明细表 / 评标办法里直接识别
- 提取结果必须包含以下典型独立项：
  - 施工方案和技术措施
  - 质量管理体系与措施
  - 安全文明施工管理体系与措施
  - 环境保护管理体系与措施
  - 工期保证措施
  - 资源配备计划
  - 施工总平面布置
  - 项目管理机构配置

#### 输出
一个字符串列表，例如：
```go
[]string{
    "施工方案和技术措施",
    "质量管理体系与措施",
    "安全文明施工管理体系与措施",
    "环境保护管理体系与措施",
}
```

---

### 任务 2：目录生成时，单独喂“必出项清单”

#### 目的
让模型明确知道哪些项不能合并、不能省略。

#### 建议位置
`backend_go/internal/service/tender_digitization.go`
中的 `GenerateOutlineDirectly`

#### 要求
把 `mandatory_chapters` 单独拼一段文本，放到 `dynamicInstruction` 里，且文案必须明确：
- 这些条目是硬性约束
- 每一项必须独立成一级章
- 不得合并到其他章节
- 宁可多一章，不可少一章

#### 文案建议
```text
【强制独立成章项（不得合并到其他章节，必须独立列为一级章）】
1. 环境保护管理体系与措施
2. 安全文明施工管理体系与措施
...

重要提示：以上每一项必须在目录中独立成为一级章（一级标题），不得合并到其他章节中。评委在一级目录中找不到对应章名，即判定为缺漏项，直接扣分。宁可多一章，不可少一章。
```

---

### 任务 3：目录生成后，做“必出项硬校验”

#### 目的
防止模型漏章后直接进入后续流程。

#### 建议位置
- `backend_go/internal/service/step4_coordinator.go`
- `backend_go/internal/handler/tech_bid_project.go`

#### 要求
在目录生成后、进入审计前，加一个硬校验步骤：
1. 获取 `mandatory_chapters`
2. 检查目录一级章里是否包含这些项
3. 如果缺失，记录日志并自动补章

#### 伪代码
```go
mandatoryChapters := ExtractMandatoryChapters(facts)
if len(mandatoryChapters) > 0 {
    missing, ok := ValidateMandatoryChapters(draftOutline, mandatoryChapters)
    if !ok {
        log.Printf("[Step4] 必出项缺失: %v，自动补章", missing)
        draftOutline = PatchMissingMandatoryChapters(draftOutline, missing)
        log.Printf("[Step4] 补章后目录共 %d 个一级章", len(draftOutline))
    }
}
```

---

### 任务 4：新增“必出项校验”函数

#### 建议位置
`backend_go/internal/service/tender_digitization.go`

#### 新增函数
1. `ExtractMandatoryChapters(facts *FactExtractResult) []string`
2. `ValidateMandatoryChapters(outline []map[string]interface{}, mandatoryChapters []string) (missing []string, ok bool)`
3. `MatchChapterWithMandatory(chapterName, mandatory string) bool`
4. `PatchMissingMandatoryChapters(outline []map[string]interface{}, missing []string) []map[string]interface{}`

#### 说明
- `ExtractMandatoryChapters`：从 facts 中筛选出必出项
- `ValidateMandatoryChapters`：检查目录是否覆盖必出项
- `MatchChapterWithMandatory`：判断章名是否命中某个必出项
- `PatchMissingMandatoryChapters`：缺章时自动补一个一级章 + 二级节 + 占位小节

---

### 任务 5：补章后必须再次校验

#### 目的
避免“补完一个坑，又出另一个坑”。

#### 要求
补章后再执行一次：
- 必出项校验
- 骨架漂移校验

#### 规则
- 通过：继续后续流程
- 不通过：回滚或重新补章

---

### 任务 6：提示词增加硬规则

#### 建议修改对象
后台提示词：`tech_bid_outline_direct_generation`

#### 追加内容
把下面这条加到提示词最后：

```text
七、评分表独立项必须独立成章
如果评分标准中明确列出了以下独立评分维度：
- 施工方案和技术措施
- 质量管理体系与措施
- 安全文明施工管理体系与措施
- 环境保护管理体系与措施
- 工期保证措施
- 资源配备计划
- 施工总平面布置
- 项目管理机构配置

则每一项必须在目录中独立成章（一级章），不得合并到其他章节中。
评委在一级目录里找不到对应章名，即判定为缺漏项，直接扣分。
因此：宁可多一章，不可少一章。
```

## 三、实施顺序

### 第一阶段：代码硬校验
优先做：
1. `ExtractMandatoryChapters`
2. `ValidateMandatoryChapters`
3. `PatchMissingMandatoryChapters`
4. 主链路里插入必出项检查

### 第二阶段：提示词补强
再做：
5. 给 `tech_bid_outline_direct_generation` 追加第七条

### 第三阶段：回归验证
最后做：
6. 用同一份招标文件重新跑目录
7. 检查“环境保护”是否独立成章
8. 检查其他独立项是否还会漏

---

## 四、验收标准

### 必须满足
1. 评分表独立项不再被合并到其他章里
2. “环境保护管理体系与措施”必须独立出现
3. 缺项会自动补章
4. 补章后会再次校验
5. 提示词和代码都明确要求“宁可多一章，不可少一章”

---

## 五、补充说明

### 当前方案的边界
- 这套方案能明显降低漏章风险
- 但它仍然依赖 facts 的识别质量
- 后续最好把“评分表必出项”进一步前移，尽量直接从评分表原文抽取，而不是完全依赖 facts

### 下一步建议
如果这次通过，再继续做两件事：
1. 把“评分表原文抽取必出项”单独前置成一个步骤
2. 把必出项和一级章建立更严格的结构化映射

---

## 六、给程序员的执行口令

**先做代码硬校验，再补提示词。**

不要只改提示词，不要只做日志提醒。  
必须做到：
- 提取
- 校验
- 补章
- 再校验

这样才算真正解决问题。
