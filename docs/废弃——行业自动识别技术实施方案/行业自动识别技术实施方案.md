# 行业自动识别技术实施方案

## 一、问题定义

当前系统在"目录生成"阶段才匹配行业骨架库，且行业分类完全依赖用户在创建项目时手动选择的"工程大类"字段。

这会导致：
- 用户选错 → 整条链路跑偏
- 项目名称写的是"天然气安装"，但系统当成"房建"处理
- 错误无法被后续步骤纠正

## 二、目标

1. **去掉用户手选"工程大类"作为主依据**
2. **系统在第一步（招标解析）自动识别行业**
3. **第二步（项目画像）确认并锁定行业分类**
4. **第四步（目录生成）直接消费已锁定的行业分类**
5. **用户只负责确认/纠正，不负责先判断**

## 三、总体设计

```
第一步 招标解析
  ↓
  解析招标文件内容
  ↓
  自动识别行业分类（初判）
  ↓
第二步 项目画像
  ↓
  结合资质要求、专业关键词、项目特征
  ↓
  确认行业分类（终判）
  ↓
  用户可见"系统识别结果"，可手动纠正
  ↓
  锁定行业分类，写入项目记录
  ↓
第四步 目录生成
  ↓
  直接读取已锁定的行业分类
  ↓
  匹配骨架库，生成目录
```

## 四、具体实施

### 4.1 第一步：招标解析阶段 → 行业初判

#### 要做什么
在招标文件解析完成后，增加一个"行业自动识别"环节。

#### 识别依据（按优先级）
1. **项目名称关键词**
   - "天然气" / "燃气" → 燃气工程
   - "市政" / "道路" / "管网" → 市政工程
   - "水利" / "水库" / "河道" → 水利工程
   - "房建" / "住宅" / "站房" → 房建工程
   - "公路" / "桥梁" / "隧道" → 公路工程

2. **招标文件正文关键词**
   - 扫描前 5000 字
   - 匹配行业特征词频
   - 取命中最多的行业

3. **资质要求**
   - "市政公用工程施工总承包" → 市政
   - "房屋建筑工程施工总承包" → 房建
   - "燃气燃烧器具安装" → 燃气
   - "水利水电工程施工总承包" → 水利

#### 输出
```json
{
  "auto_detected_industry": "燃气工程",
  "detection_source": "project_name + tender_content + qualification",
  "confidence": 0.92,
  "evidence": [
    "项目名称包含'天然气安装'",
    "招标文件提及'燃气管道安装资质'",
    "资质要求为'燃气燃烧器具安装维修资质'"
  ]
}
```

#### 建议落地文件
- `backend_go/internal/service/tender_digitization.go`
- 或新增 `backend_go/internal/service/industry_classifier.go`

---

### 4.2 第二步：项目画像阶段 → 行业终判

#### 要做什么
在项目画像生成完成后，结合画像结果对行业分类做二次确认。

#### 确认依据
1. **项目画像中的工程特征**
   - 施工范围
   - 设备材料
   - 专项技术要求

2. **与第一步初判结果的一致性**
   - 如果一致 → 直接锁定
   - 如果冲突 → 标记为"待人工确认"

#### 输出
```json
{
  "confirmed_industry": "燃气工程",
  "confirmed_by": "auto",
  "initial_detection": "燃气工程",
  "profile_validation": "一致",
  "locked": true
}
```

#### 如果不一致
```json
{
  "confirmed_industry": null,
  "confirmed_by": "pending_human",
  "initial_detection": "燃气工程",
  "profile_validation": "与画像特征不完全一致",
  "locked": false,
  "requires_human_confirm": true
}
```

#### 建议落地文件
- `backend_go/internal/service/tender_digitization.go`
- `backend_go/internal/handler/tech_bid_project.go`

---

### 4.3 前端改造

#### 4.3.1 创建项目时
- **去掉"工程大类"必选**
- 用户只填：
  - 项目名称（必填）
  - 招标编号（选填）
- 上传招标文件后，系统自动识别行业

#### 4.3.2 项目画像页
- 新增"行业识别结果"展示区
- 展示：
  - 系统识别的行业
  - 识别依据
  - 置信度
  - 是否已锁定
- 用户可点击"纠正"修改行业分类

#### 4.3.3 目录生成页
- 展示当前使用的行业骨架名称
- 如果行业未锁定，阻止目录生成，提示用户先确认行业

#### 建议落地文件
- `frontend/src/pages/TechBidProjectWorkbench.tsx`
- `frontend/src/pages/TechBidProjectList.tsx`

---

### 4.4 后端改造

#### 4.4.1 数据库字段调整

在 `tech_bid_projects` 表新增或修改：

| 字段 | 类型 | 说明 |
|------|------|------|
| auto_detected_industry | TEXT | 系统自动识别的行业 |
| detection_confidence | REAL | 识别置信度 |
| detection_evidence_json | TEXT | 识别依据 |
| confirmed_industry | TEXT | 最终确认的行业（可能是自动或人工） |
| industry_confirmed_by | TEXT | auto / human |
| industry_locked | INTEGER | 是否已锁定 |

#### 4.4.2 行业识别服务

新增 `IndustryClassifier`：
```
输入：项目名称 + 招标文件内容 + 资质要求
输出：行业分类 + 置信度 + 证据
```

#### 4.4.3 骨架匹配逻辑修改

`LoadIndustrySkeleton()` 改为：
- 优先读取 `confirmed_industry`
- 如果没有，读取 `auto_detected_industry`
- 最后才回退到 `project_type`（兼容旧数据）

#### 4.4.4 目录生成前置校验

在 `RunOutlinePipeline()` 开头增加：
- 检查 `industry_locked` 是否为 true
- 如果未锁定，中断并提示用户先确认行业

## 五、验收标准

### 5.1 创建项目
- 用户不需要手选工程大类
- 上传招标文件后系统自动识别行业

### 5.2 招标解析完成后
- 可以看到系统识别的行业和依据

### 5.3 项目画像完成后
- 行业分类已确认或提示用户确认

### 5.4 目录生成
- 使用的是已确认的行业骨架
- 不再出现"天然气项目用铁路房建骨架"的问题

### 5.5 兼容性
- 旧项目仍然可用
- 旧数据的 `project_type` 作为最低优先级回退

---

## 六、开发排期建议

| 阶段 | 内容 | 预计工时 |
|------|------|----------|
| 第一阶段 | 后端：行业识别服务 + 数据库字段 + 骨架匹配改造 | 2~3 天 |
| 第二阶段 | 前端：创建项目改造 + 画像页展示 + 目录页校验 | 2 天 |
| 第三阶段 | 联调 + 测试 + 旧数据兼容 | 1~2 天 |

---

## 七、风险提示

1. **行业关键词库需要持续维护**
   - 第一版先覆盖常见行业
   - 后续通过实际使用反馈补充

2. **旧项目兼容**
   - 旧项目没有 `confirmed_industry`
   - 需要回退到 `project_type` 字段

3. **边界行业**
   - 有些项目可能跨行业（如"燃气管道市政工程"）
   - 这种情况应该给出多个候选，让用户选

---

# 八、程序员执行版（可直接开工）

> 下面这部分按“任务目标 / 影响文件 / 开发内容 / 验收标准”写，方便直接分配给程序员。

## 任务 E1：新增行业自动识别服务

### 任务目标
实现招标解析阶段的行业初判，不再完全依赖用户手动选择。

### 影响文件
- `backend_go/internal/service/tender_digitization.go`
- 或新增 `backend_go/internal/service/industry_classifier.go`

### 开发内容
1. 输入：
   - 项目名称
   - 招标文件正文
   - 资质要求
2. 输出：
   - `auto_detected_industry`
   - `detection_confidence`
   - `detection_evidence_json`
3. 识别规则：
   - 项目名称优先
   - 招标文件关键词次之
   - 资质要求再次之
4. 如果匹配多个行业，返回候选列表，不要强行只选一个。

### 验收标准
- 项目名称包含“天然气”的项目，自动识别为燃气相关行业
- 结果里必须能看到识别依据
- 识别结果有置信度

---

## 任务 E2：项目画像阶段做行业终判

### 任务目标
在项目画像阶段锁定最终行业分类。

### 影响文件
- `backend_go/internal/service/tender_digitization.go`
- `backend_go/internal/handler/tech_bid_project.go`

### 开发内容
1. 将行业初判结果与项目画像结果做一致性校验
2. 一致则锁定行业
3. 不一致则标记为“待人工确认”
4. 行业确认结果写入项目表

### 验收标准
- 行业能被锁定
- 冲突时不会直接进入目录生成

## 任务 E4：目录生成前强制检查行业是否已锁定

### 任务目标
防止错误行业进入目录生成阶段。

### 影响文件
- `backend_go/internal/service/step4_coordinator.go`
- `backend_go/internal/handler/tech_bid_project.go`

### 开发内容
1. 在 `RunOutlinePipeline()` 开始处检查行业状态
2. 如果未锁定：
   - 阻止目录生成
   - 返回提示用户先确认行业
3. 如果已锁定：
   - 正常进入骨架匹配

### 验收标准
- 未锁定行业时不能生成目录
- 行业错误时可在前面拦截，不跑偏到第四步才发现
