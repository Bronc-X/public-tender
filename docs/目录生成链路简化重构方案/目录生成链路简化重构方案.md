# 目录生成链路简化重构方案（8步流程兼容版）

## 一、背景

当前技术标制作中的第四步“目录生成”，内部采用的是一条较复杂的技术链路：

```text
行业识别 → 骨架匹配 → 弹性调整 → 目录生成
```

这条链路的初衷，是为了降低大模型直接生成目录时的漏项风险。

但实际运行中暴露出一个更大的问题：

- 只要前面的行业识别或骨架匹配错了
- 后面整条链路就会“方向性错误”
- 错误不是“漏一点”，而是“整套目录都错”

例如：
- 项目名称是“2026—2027年度天然气安装工程招标文件”
- 但由于项目类型被标记成“房建工程”
- 目录生成阶段加载了铁路房建骨架
- 最终输出了“铁路站房工程”目录

这说明当前问题不是“第4步不该存在”，而是：

> **第4步内部的生成引擎过度依赖行业骨架匹配，导致前置错误被放大。**

---

## 二、核心结论

## 新方案结论

**不调整现有产品的 8 步流程。**

仍然保留：
1. 招标解析
2. 项目画像
3. 路线规划
4. 目录生成
5. 终审核查
6. 正文生成
7. 风控合规
8. 导出定稿

### 真正要改的是：
**只重构第 4 步“目录生成”的内部实现方式。**

也就是把第 4 步内部从：

```text
行业识别 → 骨架匹配 → 弹性调整 → 目录生成
```

改成：

```text
招标文件 + 前三步产物 → 大模型直接生成目录 → 漏项校验 → 人工确认
```

### 原则
1. **8步产品流程不动**
2. **第4步仍然叫“目录生成”**
3. **只替换第4步的生成内核**
4. **前三步继续保留，作为第4步的辅助输入**
5. **骨架库退出主链路，降级为可选参考**

---

## 三、为什么这样改最合适

### 3.1 不推翻现有产品结构
如果直接把系统改成“5步新流程”，改动太大：
- 前端流程要改
- 用户认知要改
- 文档、培训、操作习惯都要改

这没有必要。

### 3.2 现有前三步仍然有价值
前三步并不是没用：

#### 第一步 招标解析
仍然可以提供：
- 招标文件正文
- requirement register
- 关键事实

#### 第二步 项目画像
仍然可以提供：
- 项目特征
- 施工重点
- 资质要求
- 风险点

#### 第三步 路线规划
仍然可以提供：
- 写作方向
- 重点响应策略
- 章节重心建议

这些都可以继续作为第 4 步 prompt 的辅助输入。

### 3.3 只改第4步，收益最大、代价最小
问题最核心的爆点就在第 4 步：
- 不是第 1 步错
- 不是第 2 步错
- 不是第 3 步错
- 是第 4 步把错误行业骨架当成主框架，造成目录整体跑偏

所以最优策略不是推翻 8 步，而是：

> **只把第4步从“骨架主导”改成“大模型主导”。**

---

## 四、重构后的 8 步流程关系

## 4.1 前三步不变，第4步重构

### 第一步：招标解析（保留）
作用不变：
- 解析招标文件
- 提取关键 requirement / facts

### 第二步：项目画像（保留）
作用不变：
- 提炼项目特征
- 输出施工重点、资质要求、风险点

### 第三步：路线规划（保留）
作用不变：
- 形成写作路线
- 给第4步提供目录侧重点建议

### 第四步：目录生成（重构）
新逻辑：

```text
招标文件正文 + requirement/facts + 项目画像 + 路线规划
    ↓
大模型直接生成技术标三级目录
    ↓
系统做漏项校验
    ↓
人工确认/微调
```

### 第五步：终审核查（保留）
目录确认后继续做终审，不变。

### 第六步：正文生成（保留）
基于“已确认目录”生成正文，不变。

### 第七步：风控合规（保留）
继续执行，不变。

### 第八步：导出定稿（保留）
继续执行，不变。

---

## 五、系统中哪些模块保留，哪些降级，哪些退出主链路

## 5.1 保留的模块

### 保留 1：招标解析 / facts / requirement register
这些仍然有价值，用于：
- 给第4步提供辅助上下文
- 目录生成后的漏项检查
- 正文生成与风控审核

### 保留 2：项目画像
项目画像继续存在，不删除。
它的作用从“驱动骨架匹配”转为“辅助大模型理解项目特征”。

### 保留 3：路线规划
路线规划继续存在。
它给第4步提供“目录重点响应方向”。

### 保留 4：漏项检查 / full response check
这些继续保留，但改为**后置审计层**。

### 保留 5：人工确认
人工确认继续保留，而且更加重要。

---

## 5.2 降级的模块

### 降级 1：行业骨架库
骨架库不再作为第4步目录生成的主框架。

新的定位：
- 可选参考资料
- 可作为 prompt 的补充提示
- 不允许硬绑定目录结构

### 降级 2：行业识别
行业识别不再作为目录生成主链路的前置硬条件。

新的定位：
- 可作为标签信息
- 可作为后台统计或分析字段
- 不直接控制目录结构

### 降级 3：弹性结构调整
弹性结构调整不再是默认必经步骤。

新的定位：
- 作为可选优化能力
- 用于人工不满意时进行二轮优化
- 不是目录生成主入口

---

## 5.3 退出主链路的模块

### 退出主链路 1：目录生成前的行业骨架强匹配
也就是：
- 不再以 `project_type`
- 不再以 `profession`
- 不再默认 `LoadIndustrySkeleton()`
作为第4步目录生成主入口

### 退出主链路 2：骨架主导的目录生成方式
当前这种逻辑要从主链路移除：
- 先匹配 skeleton
- 再让模型在 skeleton 框架内填充目录

这个方式可以保留为旁路模式，但不再作为默认方案。

---

## 六、第4步“目录生成”新内核设计

## 6.1 新主链路

### 输入
第4步目录生成时，模型输入按优先级：
1. 招标文件正文（主输入）
2. requirement / facts（辅助输入）
3. 项目画像（辅助输入）
4. 路线规划结果（辅助输入）
5. 行业骨架（可选参考，不默认）

### 输出
- 完整三级目录 JSON
- 覆盖招标文件中的关键要求

---

## 6.2 新 Prompt 原则

提示词要明确告诉模型：

1. 你生成的是技术标目录，不是摘要
2. 你必须以招标文件内容为唯一真相源
3. 目录必须覆盖：
   - 项目概况
   - 施工总体部署
   - 主要施工方案
   - 质量管理
   - 安全文明施工
   - 进度计划
   - 资源配置
   - 资质/人员/设备响应
   - 评分项响应
   - 强制响应项
4. 不得机械套用行业模板
5. 如果发现加分项、废标项、资质要求需要单列章节，必须显式列出

---

## 6.3 目录生成后的漏项检查

### 定位
漏项检查继续存在，但变成：

> **生成后的质量校验**

而不是：

> **生成前的框架控制**

### 检查内容
目录生成后，系统从 requirement/facts 中检查：
- 评分项是否进入目录
- 废标项是否有响应章节
- 工期节点是否有响应章节
- 安全文明施工是否覆盖
- 质量标准是否覆盖
- 人员资质要求是否有对应章节

### 输出
```json
{
  "missing_items": [...],
  "weak_items": [...],
  "risk_level": "medium",
  "requires_manual_review": true
}
```

### 作用
- 如果漏项少 → 用户直接确认
- 如果漏项多 → 用户重生目录或局部修正

- 骨架驱动目录结构

### 调整 4：支持重生和人工微调
至少支持：
- 重新生成目录
- 带补漏提示重生目录
- 用户人工改章节标题/增删章节
