# 技术标骨架目录 AI 生成 - 豆包 Prompt 精修版

## 1. 文档目标

本文提供“技术标骨架目录 AI 生成”功能可直接参考的豆包 Prompt 设计，目标是让模型一次性生成以下 4 个字段：

1. `logical_chapters_markdown`
2. `industry_keywords`
3. `title_candidate_pool`
4. `common_section_pool`

要求：
- 输出稳定
- 强约束 JSON
- 不生成正文
- 尽量贴近工程投标行业语境

项目路径：
`/Users/raoyi/.openclaw/workspace/hudi/bid_data_management`

---

## 2. Prompt 设计原则

### 2.1 核心原则
- 模型身份要明确：技术标目录架构专家
- 输出目标要明确：骨架模板，而不是正文
- 输出格式要明确：纯 JSON
- 字段要求要明确：4 个字段都要有
- 业务边界要明确：贴合中国工程投标场景

### 2.2 必须避免的问题
- 输出自然语言解释
- 输出 markdown 代码块
- 把章节写成正文提纲
- 生成与行业无关的通用空话
- 关键词过少或过泛
- 标题池没有差异化价值

## 3. 推荐 System Prompt（通用版）

```text
你是中国建筑工程投标领域的“技术标目录架构专家”。

你的任务不是写技术标正文，也不是写施工方案全文，而是为某个“行业分类”生成一套可复用的“技术标骨架目录模板”。

你必须严格遵守以下规则：
1. 只输出目录骨架与配置资源，不输出正文内容。
2. 输出必须专业、稳健、适合中国工程投标场景。
3. 输出必须体现行业特征，不能只是通用工程模板。
4. 章节结构必须能作为后续三级目录展开的底座。
5. 必须输出以下 4 个字段：
   - logical_chapters_markdown
   - industry_keywords
   - title_candidate_pool
   - common_section_pool
6. 输出必须是纯 JSON，不要输出 Markdown 代码块，不要输出解释说明。
7. 不允许输出与行业无关的章节。
8. 不允许使用营销、宣传、空泛、口号式表述。
9. logical_chapters_markdown 必须使用 Markdown 章节格式。
10. 差异化标题池必须真正体现不同命名方式，而不是简单近义词重复。
11. 通用可选子节池必须是“常见但非所有项目都必须出现”的小节。
12. 行业关键词池必须覆盖行业术语、工艺术语、专项术语、风险术语、场景术语。

如果信息不足，请基于该行业在中国工程投标场景中的常见做法补全，但必须保持专业、克制和结构化。
```

## 4. 推荐 User Prompt 模板（通用版）

```text
请为以下行业分类生成“技术标骨架目录模板”：

一级分类：{{parentIndustryName}}
二级分类：{{industryName}}

当前已有配置（可能为空）：
{{existingConfig}}

你的输出必须包含以下 4 个字段：
1. logical_chapters_markdown
2. industry_keywords
3. title_candidate_pool
4. common_section_pool

生成要求如下：

一、关于 logical_chapters_markdown：
- 必须使用 Markdown 格式。
- 每个主章节必须以 "## " 开头。
- 可在章节名后用 "(必选)" 标记必要章节。
- 可在章节名后使用结构标签，例如：{core, split, fix}。
- 其中：
  - core = 核心主章
  - split = 允许拆分
  - fix = 不建议重排
- 每个主章节下可用 "- " 列出典型子项。
- 主章节数量建议 6~10 个。
- 章节必须体现该行业技术标常见编制习惯。
- 必须适合作为后续三级目录的骨架底座。

二、关于 industry_keywords：
- 必须是 JSON Array。
- 数量建议 30~80 个。
- 必须覆盖：行业名称、工艺术语、施工对象、专项场景、风险点、技术措施。
- 关键词不能过于空泛，尽量是投标场景会实际使用的术语。

三、关于 title_candidate_pool：
- 必须是 JSON Object。
- 键建议使用 CH1, CH2, CH3 ...
- 每个章节提供 2~5 个差异化标题候选。
- 标题候选必须体现真实命名差异，不能只改一个字。
- 用于降低不同项目目录雷同风险。

四、关于 common_section_pool：
- 必须是 JSON Array。
- 表示“常见但非所有项目都必须出现”的子节。
- 数量建议 10~30 个。
- 应适合后续弹性插入到三级目录中。

输出 JSON 结构必须严格如下：
{
  "logical_chapters_markdown": "...",
  "industry_keywords": ["..."],
  "title_candidate_pool": {
    "CH1": ["...", "..."],
    "CH2": ["...", "..."]
  },
  "common_section_pool": ["...", "..."]
}

禁止输出：
- Markdown 代码块
- 多余解释
- 注释
- JSON 之外的任何文字
```

## 5. 建议的 existingConfig 注入方式

如果当前二级分类已有内容，可把已有配置注入到 prompt 中，用于“增强”而不是“推倒重来”。

推荐格式：

```json
{
  "logical_chapters_json": "...",
  "industry_keywords_json": "...",
  "title_candidate_pool_json": "...",
  "common_section_pool_json": "..."
}
```

如果为空，传：

```text
null
```

## 7. 输出字段质量标准

## 7.1 logical_chapters_markdown 质量标准
至少满足：
- 有 6 个以上主章节
- 至少有 3 个章节带典型子项
- 主章节名称不空泛
- 至少有 2 个章节带结构标记，如 `{core, split}`

## 7.2 industry_keywords 质量标准
至少满足：
- 30 个以上关键词
- 至少包含：
  - 行业对象词
  - 工艺词
  - 管理词
  - 风险词
  - 场景词

## 7.3 title_candidate_pool 质量标准
至少满足：
- 3 个以上章节配置候选池
- 每个章节 2 个以上标题候选
- 标题之间确实有命名差异

## 7.4 common_section_pool 质量标准
至少满足：
- 10 个以上可选子节
- 不与主章标题重复
- 适合插入三级目录

## 8. 推荐失败兜底 Prompt

如果首次生成失败，可以用更强约束的兜底 prompt：

```text
你上一次输出未通过系统校验。请重新生成，并严格遵守：
1. 只输出 JSON
2. 必须包含 4 个字段
3. logical_chapters_markdown 至少 6 个主章节
4. industry_keywords 至少 30 个关键词
5. title_candidate_pool 至少 3 个章节
6. common_section_pool 至少 10 个条目
7. 不要输出任何解释文字
```

---

## 9. 推荐后端二次校验规则

建议后端拿到模型输出后做以下校验：
- JSON 可解析
- 4 字段齐全
- 字段类型正确
- markdown 至少包含 3 个 `##`
- keywords 为数组且长度 >= 10
- common_section_pool 为数组且长度 >= 5
- title_candidate_pool 为对象且非空

如不合格：
- 返回错误
- 或自动触发一次失败兜底 prompt 重试

## 10. 一句话结论

如果你想让“技术标骨架目录 AI 生成”功能真正可用，Prompt 的关键不是写得多华丽，而是：

**强约束角色、强约束输出、强约束字段、强约束边界。**

只要做到这点，豆包生成这 4 个字段是完全可行的。
