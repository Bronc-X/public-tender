# 后链路防污染实施方案

> CTO 审核文档 | 2026-04-10

## 一、问题定义

目录生成后，经过 审计 → 补全/优化 → 终审 链路，原始高质量目录可能被改差。  
核心原因：后链路步骤允许"整篇重写"，没有对主骨架做保护。

---

## 二、当前后链路流程（现状）

```
GenerateOutlineDirectly → draftOutline
       ↓
ValidateOutlineSemanticCoverage（本地规则校验，不调 LLM，✅ 安全）
       ↓
ValidateFullRequirementResponse（本地规则校验，不调 LLM，✅ 安全）
       ↓
AuditOutlineCoverage（调 LLM，输出缺失/薄弱项清单）
       ↓
DecideNextFlowWithFullResponse → PASS / REVISE / BLOCK
       ↓
如果 REVISE → OptimizeOutlineByCoverage（调 LLM，⚠️ 风险点：整篇重写目录）
       ↓
最终落库 → tech_bid_chapter_plans
```

### 风险点分析

| 步骤 | 调 LLM | 风险等级 | 风险描述 |
|------|--------|---------|---------|
| ValidateOutlineSemanticCoverage | ❌ | 无 | 纯本地规则，不改目录 |
| ValidateFullRequirementResponse | ❌ | 无 | 纯本地规则，不改目录 |
| AuditOutlineCoverage | ✅ | 低 | 只输出审计报告，不改目录本身 |
| OptimizeOutlineByCoverage | ✅ | **高** | 让 LLM 重新输出完整目录 JSON，可能大改主骨架 |

**结论：唯一的高风险点是 `OptimizeOutlineByCoverage`**，它会让模型重新生成整篇目录，导致一级章和二级节被改动。

---

## 三、实施方案

### 方案 1：冻结主骨架（推荐，改动最小）

**核心思想**：优化前先锁定一级章和二级节，优化后强制恢复。

#### 改动位置
`step4_coordinator.go` 中调用 `OptimizeOutlineByCoverage` 的前后

#### 具体实现

```go
// === 优化前：提取原始骨架 ===
frozenSkeleton := extractSkeleton(draftOutline) 
// frozenSkeleton = [{name: "第一章 xxx", units: [{name: "第一节 xxx"}]}]
// 只保留一级章名和二级节名，不含三级小节

// === 调用优化 ===
finalOutline, optErr := d.OptimizeOutlineByCoverage(...)

// === 优化后：强制恢复主骨架 ===
if optErr == nil {
    finalOutline = enforceSkeleton(finalOutline, frozenSkeleton)
    // 规则：
    // 1. 一级章的名称和顺序必须和 frozenSkeleton 一致
    // 2. 二级节的名称和顺序必须和 frozenSkeleton 一致
    // 3. 三级小节允许增删改（这是优化的主要空间）
    // 4. 如果优化后新增了章/节（frozenSkeleton 里没有的），追加到末尾
}
```

#### 需要新增的两个函数

```go
// extractSkeleton 提取一级章和二级节骨架
func extractSkeleton(outline []map[string]interface{}) []SkeletonEntry

// enforceSkeleton 强制恢复主骨架，只保留三级小节的变化
func enforceSkeleton(optimized []map[string]interface{}, frozen []SkeletonEntry) []map[string]interface{}
```

#### 优点
- 改动量极小（只在 coordinator 里加两行 + 两个纯函数）
- 不影响审计逻辑
- 不影响现有提示词
- 向后兼容

---

### 方案 2：强化优化提示词约束

**核心思想**：在 `OptimizeOutlineByCoverage` 的提示词里加硬约束。

#### 改动位置
`tech_bid_outline_refine` 提示词 + `OptimizeOutlineByCoverage` 的 dynamicInstruction

#### 具体实现

在 dynamicInstruction 里追加：

```
【硬性约束】
1. 严禁修改一级章（第X章）的名称和顺序
2. 严禁修改二级节（第X节）的名称和顺序
3. 只允许在现有章/节下增加、修改、删除三级小节
4. 如果审计发现的缺失项无法归入现有章节，可在末尾新增章节，但必须在 JSON 注释中说明理由
5. 如果优化后的一级章数量或名称与原始目录不一致，视为无效输出
```

#### 优点
- 不改代码逻辑
- 纯提示词层面约束

#### 缺点
- 依赖 LLM 遵守指令，不是 100% 可靠
- qwen3.5-plus 可能仍然偶尔违反

---

### 方案 3：优化结果 diff 校验 + 自动回滚

**核心思想**：优化后和原始版本做 diff，如果主骨架变化超阈值，自动回滚到原始版本。

#### 改动位置
`step4_coordinator.go` 中调用 `OptimizeOutlineByCoverage` 之后

#### 具体实现

```go
finalOutline, optErr := d.OptimizeOutlineByCoverage(...)
if optErr == nil {
    chapterDrift := computeChapterDrift(draftOutline, finalOutline)
    // chapterDrift = 一级章名称/顺序变化比例
    
    if chapterDrift > 0.3 { // 超过 30% 的章被改了
        log.Printf("[Step4] ⚠️ 优化后骨架偏移过大 (drift=%.2f)，回滚到原始目录", chapterDrift)
        finalOutline = draftOutline // 回滚
        // 记录回滚事件到日志
    }
}
```

#### 优点
- 兜底保护，不依赖 LLM 遵守指令
- 改动很小

#### 缺点
- 只是兜底，不能主动约束优化方向

---

## 四、推荐组合

**方案 1 + 方案 3 组合使用**，即：

1. 优化前提取骨架
2. 优化后强制恢复骨架（方案 1）
3. 再做 drift 校验，如果恢复后仍然偏移过大，直接回滚（方案 3）

方案 2 的提示词约束可以作为**锦上添花**，但不作为主要保护手段。

## 五、改动范围汇总

| 文件 | 改动内容 | 改动量 |
|------|---------|-------|
| `step4_coordinator.go` | 在 REVISE 分支前后加骨架冻结 + drift 校验 | ~30 行 |
| `tender_digitization.go` | 新增 `extractSkeleton` + `enforceSkeleton` + `computeChapterDrift` 三个纯函数 | ~80 行 |
| `tech_bid_project.go` | `continueOutlineGenerationFromApprovedPlan` 同样加骨架冻结 | ~15 行 |
| 提示词（可选） | `tech_bid_outline_refine` 加硬约束 | 5 行文字 |

**总改动量：约 125 行代码**  
**风险等级：低**（全部是追加逻辑，不修改现有流程）

---

## 六、不改的部分

| 步骤 | 理由 |
|------|------|
| AuditOutlineCoverage | 只输出审计报告，不改目录，无风险 |
| ValidateOutlineSemanticCoverage | 纯本地规则，无风险 |
| ValidateFullRequirementResponse | 纯本地规则，无风险 |
| DecideNextFlow | 纯逻辑判断，无风险 |
| 落库逻辑 | 不涉及目录内容变化 |

---

## 七、验证方式

1. 用同一个招标文件跑两次目录生成
2. 对比：
   - 原始生成的 draftOutline（第一口输出）
   - 经过优化后的 finalOutline
3. 检查一级章和二级节是否被保留
4. 检查三级小节是否有合理增补
