# 招标文件“项目画像抽取”技术实施方案（P0/P1/P2）

## 一、目标定义

### 1.1 业务目标
解决第二步“项目画像”中 AI 抽取不全、不稳定、不可追溯的问题，使其能够稳定支撑后续：
- 路线规划
- 目录生成
- 正文生成
- 风控审查
- 人工复核

### 1.2 技术目标
把当前“单次 prompt 抽取”升级为“可观测、可审计、可补漏”的结构化抽取链路。

### 1.3 成功标准
第一阶段完成后，至少做到：
- 关键字段稳定输出
- 每个关键字段可追溯到原文证据
- 长文档分块后不因 merge 丢失关键信息
- 能显式告诉用户“哪里没抽到 / 哪里不确定”
- 前端能看到缺失项与低置信度提醒

---

## 二、实施原则

1. 先修链路，再卷 prompt
2. 高风险字段优先，不做大而全
3. 所有结果必须可追溯
4. 分阶段上线，不一次性重构
5. 每阶段必须有验收指标

---

## 三、问题拆解

当前“项目画像抽不全”可能来自 5 层：

1. 文档解析层
   - OCR / PDF / Word 文本提取不完整
   - 表格、附录、清单丢失

2. 分块层
   - chunk 切分破坏上下文
   - 表格和正文混切，导致语义损坏

3. 抽取层
   - prompt 不够约束
   - schema 不固定
   - 模型把关键信息塞进摘要，没进结构字段

4. 合并层
   - chunk 结果 merge 逻辑过于简单
   - 后值覆盖前值，导致信息丢失

5. 展示/消费层
   - 后端明明抽到了，前端没显示
   - 前端没提示缺失、不确定、冲突

---

## 四、总体架构（目标态）

### 4.1 抽取链路
```text
文档预处理
  ↓
块识别（正文 / 表格 / 附录 / 清单）
  ↓
主抽取（固定 schema）
  ↓
高风险专项补漏
  ↓
关键词反查审计
  ↓
字段级 merge
  ↓
缺失/冲突/低置信度标记
  ↓
前端展示 + 人工复核
  ↓
下游写作模块消费
```

### 4.2 输出形态
最终输出不再是“长摘要”，而是：
- 总画像（总览）
- 专项画像（采购、工期、质量、安全、评分、资格）
- 审计结果（缺失、不确定、人工复核）
- 证据链（source_text/source_location）

# 五、P0：先止血，建立最小可用闭环（必须先做）

> 目标：2~5 天内能落地，快速提升“项目画像可用性”。

## P0-1：建立链路可观测性

### 要做什么
在每个处理阶段落日志和中间产物，至少保存：
- 原始解析文本
- chunk 列表
- 每个 chunk 的抽取结果
- merge 前结果
- merge 后结果
- 前端最终响应结果

### 验收标准
随机抽 3 份文档，能够追溯任意一个字段从：
原文 → chunk → 提取 → merge → 前端 的完整路径。

## P0-2：固定最小可用 Schema

### 推荐第一版 schema
```json
{
  "project_base_info": {
    "project_name": {},
    "project_type": {},
    "project_location": {},
    "contract_scope_summary": {},
    "total_duration": {},
    "quality_target": {},
    "safety_target": {}
  },
  "construction_core_requirements": {
    "construction_scope": {},
    "material_equipment_rules": {},
    "procurement_boundary": {},
    "owner_supplied_items": [],
    "contractor_supplied_items": [],
    "quality_acceptance_rules": {},
    "safety_civilization_rules": {},
    "schedule_constraints": {}
  },
  "bidder_requirements": {
    "qualification_requirements": {},
    "personnel_requirements": {},
    "performance_requirements": {},
    "certificate_requirements": {}
  },
  "evaluation_and_performance_rules": {
    "scoring_items": [],
    "bonus_rules": [],
    "deduction_rules": [],
    "disqualification_rules": [],
    "mandatory_response_items": []
  },
  "extraction_gaps": [],
  "uncertain_items": [],
  "requires_manual_review": []
}
```

### 规则
- 所有字段必须输出
- 没有也要输出空结构
- 不允许自由新增字段
- 不允许把关键字段藏进摘要

## P0-3：每个字段强制带证据

### 标准结构
```json
{
  "value": "",
  "source_text": [],
  "source_location": [],
  "confidence": 0,
  "missing": false,
  "notes": ""
}
```

### 验收标准
高风险字段至少 80% 带证据；空字段明确 missing=true。

## P0-4：增加“缺失 / 不确定 / 人工复核”输出

### 新增三个统一区域
```json
{
  "extraction_gaps": [],
  "uncertain_items": [],
  "requires_manual_review": []
}
```

### 验收标准
不能再出现“看起来很完整，实际上漏很多”的假完整输出。

## P0-5：先补 4 类高风险专项补漏

### 第一批专项补漏
1. 采购边界
2. 工期节点
3. 评分与废标
4. 资格与人员

### 验收标准
抽样文档中，这 4 类字段召回率明显提升；至少能发现漏项并进入 requires_manual_review。

## P0-6：前端显示缺失和低置信度标签

### 标签
- 已提取
- 缺失
- 低置信度
- 待人工确认
- 多候选冲突

### 验收标准
用户在页面上能一眼看出哪些字段已稳定抽到，哪些字段需要人工确认。

# 六、P1：补强抽取能力，解决“分块与表格漏项”问题

> 目标：1~2 周，解决长文档、表格、附录、merge 丢信息问题。

## P1-1：重写 merge 逻辑为字段级 merge

### 新 merge 规则
- 对象字段：递归 merge
- 字符串字段：保留更完整、更具体版本
- 数组字段：按语义去重并保留证据
- 证据字段：一律 append 合并，不允许覆盖
- 高风险字段：保留 value + candidates + evidence_list

## P1-2：正文 / 表格 / 附录 / 清单分流抽取

### 处理方式
- 正文块 → 综合规则抽取
- 表格块 → 表格专项抽取
- 附录块 → 约束补充抽取
- 清单块 → 设备/评分/资格专项抽取

## P1-3：关键词反查审计

### 第一批关键词组
- 采购边界：甲供 / 乙供 / 招标人采购 / 中标人采购 / 自行采购 / 供货范围 / 责任界面
- 工期：工期 / 节点 / 里程碑 / 完工 / 交付 / 延误 / 奖罚
- 质量：验收 / 检验 / 试验 / 规范 / 标准 / 合格率
- 安全：安全 / 文明施工 / 环保 / 危大工程 / 专项方案
- 评分废标：评分 / 加分 / 扣分 / 否决 / 废标 / 不得分 / 必须响应

### 规则
如果原文命中关键词，但结构字段为空：自动进入 requires_manual_review 并前端标红。

## P1-4：新增专项画像层

### 建议拆分
- 总画像
- 材料设备专项画像
- 工期专项画像
- 质量专项画像
- 安全文明专项画像
- 评分专项画像
- 风险专项画像

## P1-5：前端增加人工补录入口

### 能力
- 手工补录
- 修改确认
- 标记“人工已确认”

# 七、P2：系统化增强，建立长期可持续能力

> 目标：2~4 周，建立回归、规则引擎、行业增强能力。

## P2-1：规则引擎补强

### 第一批规则能力
- 正则识别工期天数、节点日期
- 正则识别资质等级、证书名称
- 正则识别评分分值、权重、加减分
- 正则识别甲供/乙供/招标人采购等采购边界词
- 正则识别人员数量与岗位要求

## P2-2：建立最小回归测试集

### 样本覆盖
- 不同行业
- 不同排版质量
- 有正文、表格、附录、清单
- 有采购边界、评分、工期、资格要求

### 核心指标
- 高风险字段召回率
- 证据可追溯率
- 漏项预警命中率
- 人工修正率

## P2-3：保存 chunk 级原始抽取结果

### 需要保存
- 每个 chunk 的原始结构化结果
- 每个 chunk 的证据列表
- merge 变更记录

## P2-4：行业词典与领域增强

### 行业优先级
- 能源燃气
- 市政
- 水利
- 房建

## P2-5：多模型交叉审计（可选）

### 适用字段
- 废标项
- 评分标准
- 采购边界
- 工期节点

# 十、验收标准

## P0 验收
- 固定 schema 稳定输出
- 高风险字段有证据链
- 页面可看到缺失/低置信度
- 4 类高风险字段明显改善

## P1 验收
- 长文档分块后信息不再因 merge 丢失
- 表格/附录关键内容能进结构字段
- 关键词命中但字段为空时能自动预警
- 用户可人工补录并参与下游消费

## P2 验收
- 有可持续回归体系
- 行业场景抽取效果可稳定提升
- 高风险字段漏项率持续下降

---

# 十一、 CTO 版最终建议

这次不要做成“全面重构”，要做成：

**第一阶段先止血，第二阶段补结构，第三阶段做护城河。**

一句话落地策略：

> 用最小 schema + 证据链 + 高风险补漏 + 字段级 merge + 前端复核，先把项目画像从“看起来有结果”变成“真的能用于生产”。
